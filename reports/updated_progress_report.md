## 1. Executive Summary
- Decision to be supported: help a simulated content-platform team decide how to monitor KPI health, diagnose abnormal performance changes, evaluate experiment tradeoffs, and now estimate which segment-day observations are more likely to deliver stronger conversion performance.
- Business value: the project provides a reusable analytics workflow that links KPI monitoring, anomaly alerting, segmented diagnosis, A/B evaluation, and a small predictive-modeling workstream in a single decision-support prototype.
- Data assets: the repository contains a simulated daily fact table at `data/raw/content_platform_mock_data.csv`, generated by `src/generate_data.py`, plus downstream tables, figures, and dashboard outputs in `reports/`.
- What has been achieved since the midterm report: the project previously showed stronger progress in packaging, appendix organization, and English presentation cleanup than in new analytical depth. The repository now goes beyond that stage by adding an interpretable logistic-regression prototype, a factor-by-factor interpretation package for the coefficient chart, a Random Forest benchmark, and a nested cross-validation workflow for fairer model comparison.

The repository remains a simulated analytics and decision-support prototype. It demonstrates how a business analytics workflow could be structured, but it is not evidence of a production deployment or a finalized business strategy.

## 2. Data Preprocessing
- Dataset description: the mock dataset represents daily platform performance across date, content category, traffic source, user type, city tier, experiment group, hour bucket, and related KPI fields.
- Dataset summary: the current repo outputs report 97,200 observations, 20 variables, 12 numeric variables, 8 categorical variables, and 1 time variable.
- Input data visualization: the repo already contains presentation-friendly charts such as `reports/figures/dashboard_overview.png`, `reports/figures/trend_monitoring.png`, `reports/figures/step5_metric_trend_with_anomalies.png`, and `reports/outputs/appendix_dashboard_report.html`.
- Data cleaning checks: the current outputs show 0 total missing values and 0 duplicate rows. Existing notebook-style checks also verify logical KPI relationships such as `click <= exposure`, `view_complete <= click`, and `conversion_cnt <= click`.
- Outlier / anomaly detection methods: the anomaly module is rule-based rather than predictive. It computes daily KPIs, uses 7-day rolling baselines and rolling standard deviations, and adds segment-level alert rules for key business slices.
- What was newly improved since the midterm: the repo includes more organized appendix outputs for preprocessing and reporting, including `reports/outputs/dataset_summary.csv`, `reports/outputs/variable_categories.csv`, and `reports/outputs/missing_values.csv`.

The preprocessing layer is therefore more presentation-ready than it was at midterm, even though the underlying data source remains simulated.

### Predictive Prototype Added in This Update
- Chosen target: `high_conversion_performance`, defined as whether a row-level `conversion_rate` is at or above the training-set median.
- Task type: binary classification.
- Main interpretable model: logistic regression.
- Benchmark comparison model: Random Forest.
- Why this was the best lightweight setup: it uses a real business-relevant outcome already supported by the dataset, keeps one model highly interpretable, and then checks whether a slightly more flexible model materially improves prediction.

### Predictive Workflow
- Initial baseline setup: time-based split, using the earlier 78% of dates for training and the latest 22% for testing.
- Improved nested-CV setup: 3 outer folds estimate generalization performance; 3 inner folds tune hyperparameters and run `SelectKBest(f_classif)` feature selection inside the model pipeline.
- Leakage control: the `high_conversion_performance` threshold is computed separately from the training rows of each outer fold, and downstream variables such as `click`, `conversion_cnt`, `view_complete`, `report_cnt`, `monetization_revenue`, and row-level `conversion_rate` are excluded from model features.
- Features used:
  - Categorical: `content_category`, `traffic_source`, `user_type`, `city_tier`, `experiment_group`, `hour_bucket`, `week_day`
  - Numeric: `content_supply_cnt`, `session_uv`, `is_weekend`
- Features intentionally excluded: direct downstream outcome fields such as `click`, `conversion_cnt`, `view_complete`, `report_cnt`, and `monetization_revenue`.

### Initial Baseline Predictive Results
- Logistic Regression: accuracy 0.6219, precision 0.6082, recall 0.7879, F1-score 0.6865, ROC-AUC 0.6606
- Random Forest: accuracy 0.6133, precision 0.6090, recall 0.7371, F1-score 0.6670, ROC-AUC 0.6536

### Nested Cross-Validation Results
- Logistic Regression average outer-fold metrics: accuracy 0.5940, precision 0.5767, recall 0.7126, F1-score 0.6372, ROC-AUC 0.6383
- Random Forest average outer-fold metrics: accuracy 0.5939, precision 0.5724, recall 0.7458, F1-score 0.6477, ROC-AUC 0.6367
- Logistic Regression best setting selected by inner CV most often: `C = 0.1`, `penalty = l1`, `SelectKBest k = 10`
- Random Forest best setting selected by inner CV most often: `n_estimators = 80`, `max_depth = 8`, `min_samples_leaf = 20`, `max_features = sqrt`, `SelectKBest k = all`

### Interpretation
- The modeling task predicts whether a segment-day observation is likely to show relatively strong conversion performance, not causal impact or absolute business value.
- The strongest positive signals in the logistic model are `traffic_source = search` and `user_type = active_user`.
- The strongest negative signal is `user_type = new_user`, which aligns with the broader descriptive pattern that new users are harder to convert efficiently.
- The factor-interpretation package now explains all ten factors shown in the logistic coefficient chart and adds compact data-backed support for each one.
- In the earlier baseline split, Logistic Regression slightly outperformed Random Forest overall. Under nested CV, Logistic Regression has slightly better average ROC-AUC, while Random Forest has slightly better average F1-score. This should be framed as a tradeoff rather than a decisive production-model winner.
- The nested-CV results provide a more reliable estimate of general predictive performance than the original single split because hyperparameter tuning and feature selection happen inside the validation process.
- These results are reasonable for a presentation-oriented prototype, but they are not production-ready and should not be treated as real-world forecasting evidence.
