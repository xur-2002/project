# Project Title
Data-Driven Decision Support for Evaluating AI-Generated Advertising Content in a Simulated Content Platform

# Authors and Team

## 1. Executive Summary
- Decision to be supported: help a simulated content-platform team decide how to monitor KPI health, diagnose abnormal performance changes, evaluate experiment tradeoffs, and now estimate which segment-day observations are more likely to deliver stronger conversion performance.
- Business value: the project provides a reusable analytics workflow that links KPI monitoring, anomaly alerting, segmented diagnosis, A/B evaluation, and one lightweight predictive module in a single decision-support prototype.
- Data assets: the repository contains a simulated daily fact table at `data/raw/content_platform_mock_data.csv`, generated by `src/generate_data.py`, plus downstream tables, figures, and dashboard outputs in `reports/`.
- What has been achieved since the midterm report: the project previously showed stronger progress in packaging, appendix organization, and English presentation cleanup than in new analytical depth. This update now adds a real but lightweight predictive modeling prototype, while preserving the cleaner submission materials and presentation-ready outputs already added after the midterm stage.

The repository remains a simulated analytics and decision-support prototype. It demonstrates how a business analytics workflow could be structured, but it is not evidence of a production deployment or a finalized business strategy.

## 2. Data Preprocessing
- Dataset description: the mock dataset represents daily platform performance across date, content category, traffic source, user type, city tier, experiment group, hour bucket, and related KPI fields.
- Dataset summary: the current repo outputs report 97,200 observations, 20 variables, 12 numeric variables, 8 categorical variables, and 1 time variable.
- Input data visualization: the repo already contains presentation-friendly charts such as `reports/figures/dashboard_overview.png`, `reports/figures/trend_monitoring.png`, `reports/figures/step5_metric_trend_with_anomalies.png`, and `reports/outputs/appendix_dashboard_report.html`.
- Data cleaning checks: the current outputs show 0 total missing values and 0 duplicate rows. Existing notebook-style checks also verify logical KPI relationships such as `click <= exposure`, `view_complete <= click`, and `conversion_cnt <= click`.
- Outlier / anomaly detection methods: the anomaly module is rule-based rather than predictive. It computes daily KPIs, uses 7-day rolling baselines and rolling standard deviations, and adds segment-level alert rules for key business slices.
- What was newly improved since the midterm: the repo includes more organized appendix outputs for preprocessing and reporting, including `reports/outputs/dataset_summary.csv`, `reports/outputs/variable_categories.csv`, and `reports/outputs/missing_values.csv`.

The preprocessing layer is therefore more presentation-ready than it was at midterm, even though the underlying data source remains simulated.

## 3. Model Update
- Models currently used or planned: the repository now includes one lightweight predictive prototype in addition to the descriptive and diagnostic modules. The active modules are KPI aggregation, rule-based anomaly detection, segmented contribution analysis, simulated A/B evaluation, and a logistic-regression classifier in `src/lightweight_predictive_model.py`.
- Machine learning workflow description (MLM): the current workflow is `generate_data.py` -> data checks / KPI summaries -> `anomaly_detection.py` -> `root_cause_analysis.py` -> `ab_test_analysis.py` -> `lightweight_predictive_model.py` -> report outputs and presentation materials.
- What is still descriptive analytics vs what is now moving toward predictive analytics:
  - Descriptive / diagnostic: KPI summaries, trend plots, anomaly flags based on fixed rules, root-cause contribution breakdowns, and experiment lift calculations.
  - Predictive analytics newly added: a binary classification prototype predicts whether a segment-day observation is likely to achieve above-median conversion performance.
  - Still planned beyond the current prototype: richer forecasting, anomaly-risk scoring, or more advanced conversion modeling if time allows.
- Clarify the role of the simulated A/B evaluation module: the A/B component remains a decision-support evaluation layer for a simulated intervention. It estimates whether variant B improves growth-related KPIs while worsening guardrail KPIs, and it should still be interpreted as an experiment-analysis prototype rather than proof of a deployed strategy.

### Predictive Prototype Added in This Update
- Chosen target: `high_conversion_performance`, defined as whether a row-level `conversion_rate` is at or above the training-set median.
- Task type: binary classification.
- Chosen model: logistic regression.
- Why this was the best lightweight option: it uses a real business-relevant outcome already supported by the dataset, keeps the workflow interpretable, and is easier to explain in a presentation than a heavier or noisier alternative.

### Predictive Workflow
- Train/test setup: time-based split, using the earlier 78% of dates for training and the latest 22% for testing.
- Features used:
  - Categorical: `content_category`, `traffic_source`, `user_type`, `city_tier`, `experiment_group`, `hour_bucket`, `week_day`
  - Numeric: `content_supply_cnt`, `session_uv`, `is_weekend`
- Features intentionally excluded: direct downstream outcome fields such as `click`, `conversion_cnt`, `view_complete`, `report_cnt`, and `monetization_revenue`.

### Predictive Results
- Accuracy: 0.6219
- Precision: 0.6082
- Recall: 0.7879
- F1-score: 0.6865
- ROC-AUC: 0.6606

### Interpretation
- The model predicts whether a segment-day observation is likely to show relatively strong conversion performance, not causal impact or absolute business value.
- The strongest positive signals in the simulated data are `traffic_source = search` and `user_type = active_user`.
- The strongest negative signal is `user_type = new_user`, which aligns with the broader descriptive pattern that new users are harder to convert efficiently.
- These results are reasonable for a simulated, presentation-oriented prototype, but they are not production-ready and should not be treated as real-world forecasting evidence.
